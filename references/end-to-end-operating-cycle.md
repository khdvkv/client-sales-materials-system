# End-to-End Operating Cycle

## Job

Carry a client sales-materials programme from evidence to verified use without skipping meaning, creative or release gates. Each phase has an input, a durable output, an acceptance criterion and a next allowed action.

## Twelve phases

| Phase | Required work | Durable output | Acceptance and next action |
| --- | --- | --- | --- |
| 1. Client evidence | Open current project files, accepted artifacts, meeting transcripts, official client sources and relevant external evidence. Separate current source, fallback and inference. | Source and claim register with date, owner, privacy and status. | Coverage is explicit; unsupported claims stay out. Then select materials. |
| 2. Material selection | Map buyer roles, moments, decisions, channels, frequency, proof readiness and next actions. | Prioritised material register and minimal sufficient pack. | Every item has a commercial job and owner. Then research each type. |
| 3. Best-in-class | Find complete current examples performing the same document job. Inspect sequence, information objects, proof order, CTA and format. | Reference receipt per material with complete links/files, transferable units and rejected archetypes. | At least three complete examples when available; otherwise `reference_gap`. Then approve the meaning plan. |
| 4. Final copy | Write final client-visible titles, body, proof, captions, qualifications and CTA. Apply source, privacy and voice checks. | Page-by-page or section-by-section client copy. | No production notes or future-content descriptions. Then visualise. |
| 5. Art direction | Build atlas, creative proposition, divergent territories and representative real-content fragment. | Direction receipt and benchmark score. | Human-approved territory and score threshold. Then produce visual assets and layouts. |
| 6. Visual production | Retouch verified client photos or generate new Image2 assets from production cards. Build diagrams and information objects. | Asset register and production-ready visuals. | Fidelity, resolution, rights, brand and factual reality checked. Then assemble. |
| 7. Brand application | Apply logo, colour, type, image treatment, grid and diagram grammar from opened client sources. | Style receipt and reusable component rules. | Every retained code is source-backed or explicitly approved. Then build formats. |
| 8. HTML and PDF | Build the authoritative editable HTML/PPTX/DOCX surface and verified PDF/print exports. | Standalone files and export receipt. | Real-surface render, type, links, page count and image checks pass. Then assemble cabinet. |
| 9. Cabinet and PP Pages | Use full client-world names, independent open/download, editing where requested, a self-contained offline copy and controlled PP Pages publication. | Client cabinet, local review HTML and publication receipt. | Same accepted content works locally, in standalone files and online. Then release QA. |
| 10. QA audits | Run source, copy, visual, format, accessibility, privacy, navigation, download and cross-surface regression checks. | QA ledger with fresh observations. | No blocking defect; Human Gate recorded. Then release. |
| 11. Process control | Show iteration, completed scope, remaining scope, internal/client readiness, next verification and decision point. | Process passport. | The next action stays within the accepted phase and authority. |
| 12. Loop learning | Observe use or review, diagnose one divergence, repair, verify protected invariants and classify learning. | Local learning receipt or scoped shared candidate with eval and rollback. | `effect_verified` requires fresh human/use evidence. |

## Evidence collection order

1. Current client and project sources in the authoritative project route.
2. Full meeting transcripts and decisions, with date and participants.
3. Existing accepted client materials and brand assets.
4. Current official client website, documents, registers and public channels.
5. External category, buyer and format evidence.
6. Adjacent project reuse only as a labelled mechanism reference.

Never import personal contacts, Paper Planes internal information, another client's facts or a previous project's claims into client-visible content.

## Human Gates

- material stack and priority;
- meaning plan per new semantic material type;
- selected visual territory and representative fragment;
- complete section or standalone material;
- complete cabinet before client release.

The reviewer may explicitly move a gate to the full-package level. Until then every unreviewed type remains `internal_prototype`.

## Stop conditions

Stop expansion when the meaning plan is missing, references perform another document job, final copy is incomplete, the visual territory fails its threshold, the latest semantic type lacks its required Human Gate, source/privacy gaps affect visible content, or two repair cycles produce no observed improvement.
